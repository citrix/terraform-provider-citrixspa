# Citrix SPA Provider

The Citrix SPA provider manages [Citrix Secure Private Access](https://www.citrix.com/products/citrix-secure-private-access/) resources through the SPA API. It is primarily designed for **migrating resources between Citrix Cloud accounts** via Terraform.

Documentation for the supported [Data Sources](https://developer.hashicorp.com/terraform/language/data-sources) and [Resources](https://developer.hashicorp.com/terraform/language/resources) can be found in the navigation to the left.

For additional details on the SPA API, refer to the [Citrix Secure Private Access API documentation](https://developer-docs.citrix.com/en-us/secure-private-access/access-security/getting-started.html).

## Getting Started

### Prerequisites

- Terraform >= 1.0
- A Citrix Cloud account with SPA service enabled
- Either a direct auth token or Service Principal credentials (see [Authentication](#authentication))

### Create a Citrix Cloud Service Principal

Follow the [Citrix Cloud API Access with Service Principals](https://developer-docs.citrix.com/en-us/citrix-cloud/citrix-cloud-api-overview/get-started-with-citrix-cloud-apis#citrix-cloud-api-access-with-service-principals) guide to create a service principal for your cloud customer.

## Authentication

The provider supports two authentication methods. Use **only one** at a time.

### Service Principal (recommended)

```hcl
provider "spa" {
  customer_id   = "your-customer-id"
  client_id     = "your-client-id"
  client_secret = "your-client-secret"
}
```

Or via environment variables:

```bash
export CITRIX_CUSTOMER_ID="your-customer-id"
export CITRIX_CLIENT_ID="your-client-id"
export CITRIX_CLIENT_SECRET="your-client-secret"
```

### Direct Auth Token

```hcl
provider "spa" {
  customer_id = "your-customer-id"
  auth_token  = "your-auth-token"
}
```

Or via environment variables:

```bash
export CITRIX_CUSTOMER_ID="your-customer-id"
export CITRIX_AUTH_TOKEN="your-auth-token"
```

## Example Usage

```hcl
terraform {
  required_providers {
    spa = {
      source = "citrix/spa"
    }
  }
}

provider "spa" {
  customer_id   = var.citrix_customer_id
  client_id     = var.citrix_client_id
  client_secret = var.citrix_client_secret
}
```

### Optional

- `base_url` (String) Base URL for the SPA API. Should include the `/accessSecurity` path. Default: `https://api.cloud.com/accessSecurity`.
- `token_url` (String) Authentication URL to get the access token using the Service Principal authentication option. Defaults to the base URL scheme and host.
- `customer_id` (String) Citrix Cloud Customer ID. Can also be set with the `CITRIX_CUSTOMER_ID` environment variable.
- `auth_token` (String, Sensitive) Citrix Cloud authorization token (CWSAuth Bearer format). Alternative to `client_id`/`client_secret`. Can also be set with the `CITRIX_AUTH_TOKEN` environment variable.
- `client_id` (String, Sensitive) Citrix Cloud Service Principal Client ID. Used with `client_secret` for OAuth2 authentication. Can also be set with the `CITRIX_CLIENT_ID` environment variable.
- `client_secret` (String, Sensitive) Citrix Cloud Service Principal Client Secret. Used with `client_id` for OAuth2 authentication. Can also be set with the `CITRIX_CLIENT_SECRET` environment variable.
- `rate_limit` (Number) Rate limit in requests per second. Default is `15` (900 requests per minute). Set to `0` to use the default, or a negative value for unlimited.
- `fetch_details_on_list` (Boolean) When `true`, automatically fetch detailed information for each resource during list operations. This makes additional API calls. Default: `false`.
- `enable_token_cache` (Boolean) Enable encrypted disk-based token caching to reuse authentication tokens across Terraform operations. Tokens are encrypted using AES-256-GCM and stored in `~/.terraform.d/spa-cache/`. Default: `true`.

## Migrating Resources Between Accounts

The primary use case for this provider is migrating an SPA configuration from one Citrix Cloud customer account to another. The steps below walk through the full migration process.

### Step 0 — Prerequisites

- Install [Terraform](https://developer.hashicorp.com/terraform/downloads) on your machine.
- If the `citrix/spa` provider is not available in the public registry for your architecture, download the provider binary zip for your platform and extract it into the Terraform plugin directory:

  ```powershell
  # Example for Windows AMD64 — adjust the path and version as needed
  $pluginDir = "$env:APPDATA\terraform.d\plugins\registry.terraform.io\citrix\spa\<version>\windows_amd64"
  New-Item -Path $pluginDir -ItemType Directory -Force
  Expand-Archive "terraform-provider-spa_<version>_windows_amd64.zip" -DestinationPath $pluginDir -Force
  ```

  Then create (or update) `C:\Users\<you>\.terraformrc` so Terraform uses the local copy:

  ```hcl
  provider_installation {
    filesystem_mirror {
      path    = "C:/Users/<you>/AppData/Roaming/terraform.d/plugins"
      include = ["citrix/spa"]
    }
    direct {
      exclude = ["citrix/spa"]
    }
  }
  ```

### Step 1 — Create Working Folders

Create two separate folders — one for the **source** customer and one for the **destination** customer. For example:

```
migration/
  customerA/   ← source account
  customerB/   ← destination account
```

### Step 2 — List Resources in the Source Account

Inside the **source** customer folder, create a `terraform.tfvars` file with the source account credentials:

```hcl
customer_id   = "<source-customer-id>"
client_id     = "<source-client-id>"
client_secret = "<source-client-secret>"
```

Then run `spa_manager.ps1` with the `-List` flag to discover all SPA resources and generate the Terraform configuration files:

```powershell
cd migration/customerA
spa_manager.ps1 -List
```

This generates `spa_resources.tf` (resource definitions) and `provider.tf` in the source folder.

### Step 3 — Copy Configuration to the Destination Folder

Copy the generated `spa_resources.tf` and `provider.tf` files from the source folder into the **destination** customer folder:

```powershell
Copy-Item customerA/spa_resources.tf customerB/
Copy-Item customerA/provider.tf       customerB/
```

Also make sure that you have a `terraform.tfvars` file with the destination account credentials the **destination** customer folder, similar to the one in the **source** customer folder.

### Step 4 — Update IDs and User References

Open `spa_resources.tf` in the destination folder and perform the following find-and-replace operations:

1. **Replace the source customer ID** with the destination customer ID everywhere it appears.
2. **Replace resource location UIDs** (connector/location IDs) with the corresponding UIDs from the destination account resource locations.
3. **Update access policy user/group identifiers** as needed — this typically means replacing source-account OID references with destination-account equivalents, and/or adding email-based identifiers if the source customer has not been migrated to CEP but the destination customer has

### Step 5 — Apply the Configuration

Initialise Terraform and apply the configuration in the destination folder:

```powershell
cd migration/customerB
terraform init
terraform apply
```

Review the proposed changes in the plan output, then type `yes` and press **Enter** to confirm.

### Step 6 — Re-run on Errors

If `terraform apply` reports errors (for example due to transient API issues), simply re-run it:

```powershell
terraform apply
```

Terraform is idempotent — resources that were already created successfully will be skipped, and only the failed ones will be retried.

-> **Note** `spa_manager.ps1` generates resources with all inter-dependencies already embedded, so no manual `depends_on` management is needed.
